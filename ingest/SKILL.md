# ingest 技能

## 核心工作流：Inbox & Archive

你正在维护一个 **LLM Wiki**（Obsidian 知识库）。`raw/` 目录是"待处理收件箱"，`wiki/` 是"编译输出层"。

**目录结构约定：**
- `raw/01-日常/` — 日常记录的 Markdown 文章
- `raw/02-投资/` — 投资相关动态、思考、复盘的 Markdown 文章
- `raw/03-工作/` — 工作相关的 Markdown 文章
- `raw/04-Clippings/` — 剪藏的 Markdown 文章
- `wiki/sources/` — 资料摘要
- `wiki/entities/` — 实体（人物、公司、工具、产品）
- `wiki/concepts/` — 概念（框架、方法论、理论）

## 元指令（必须严格遵守）

1. **禁止模拟**：本技能是确定性流水线。每一步都必须实际执行（Read/Write/Glob），**严禁**在脑海中模拟结果或编造内容。
2. **禁止跳过**：步骤 6 的验证是强制检查点，未通过验证必须回滚。
3. **禁止修改源文件**：`raw/` 目录下的任何文件都不得被修改。
4. **禁止跳步**：严格按照步骤0→1→2.1→2.2→2.3→3→4→5→6→7的顺序执行，**不得跳过任何步骤**。每个步骤开始前必须声明：`[步骤X] 正在执行：[步骤名称]`。
5. **禁止跳跃声明**：`[步骤X]` 声明必须与实际执行的步骤严格对应。若发现声明了 `[步骤2.1]` 但实际执行的是 `[步骤3]` 的内容，视为严重错误，**立即终止并回滚**。

## 触发逻辑

1. **用户执行 `/ingest`**：扫描 `raw/` 所有子目录，找出待处理文件，按修改时间从旧到新排序，每次仅处理队列中的第一个文件。
2. **用户执行 `/ingest <path>`**：处理指定的文件或目录。
3. **隐式触发**：用户提及"摄入知识库"、"导入资料"等意图时，自动执行。

### 路径解析规则
- **默认模式（Strict）**：`<path>` 必须是相对于仓库根目录的完整路径（如 `raw/02-投资/xxx.md`）。路径不完整或无法匹配 → **直接报错终止**。
- **宽松模式（Opt-in）**：仅当用户指令包含"模糊"、"搜索"、"找一下"时，才在 `raw/` 下进行模糊搜索。若匹配到多个文件，列出候选列表请求用户确认；若匹配到 0 个，报错终止。
- **目录处理**：如果 `<path>` 是目录，先扫描统计文件数量，然后通过循环逐个处理目录内的文件（遵循一次处理一个的原则）。

1. 唯一性：能用一句话唯一指向某个具体事物 → 实体

1. 显性概念：文中直接提及的名称性概念（如"二八定律"、"断板"）
2. 隐性概念：作者在文中论证后得出的结论性判断，虽未命名但可提炼为独立概念
识别触发点：①对比性表述（"表面…实际…"、"看似…实则…"）；②反问句（"我是否只是…"）；③情绪/策略转折点（"信念重建"、"正反馈验证"）

## 提取要求
- 不仅提取文中明确命名的概念
- 还要提炼"作者暗示但未直接说出口"的结论性概念

- 示例：文中说"新标的接过接力棒冲击涨停，说明逻辑被市场接受"，可提炼出"信念重建"或"策略正反馈验证"

### 步骤 5：股票数据富化

**[步骤5] 正在执行：股票数据富化**

**步骤进度追踪**：在完成步骤5后，在内部记录 `步骤5已完成=是`

**前置条件：** 步骤2中提取的实体包含上市公司名称或股票代码（如 603629、利通电子、腾讯控股）。

**市场支持：** A股（sh/sz）、港股（hk）。美股暂不支持（仅支持中文名搜索）。

**执行流程：**

#### 4.1.1 代码与名称补全
- 如果实体只提供**名称**而无代码：
  - 调用 `mcp__aktools__search(keyword=公司名)` 尝试补全（不指定 market，由返回值推断）
  - 若返回 `code`，根据 code 前缀推断市场（SH→sh, SZ→sz, HK→hk）
  - 若返回空或失败，换用 `mcp__aktools__search(keyword=公司名, market=sh)` 重试
  - 若仍失败，换用 `mcp__aktools__search(keyword=公司名, market=sz)` 重试
  - 若仍失败，换用 `mcp__aktools__search(keyword=公司名, market=hk)` 重试
- 如果实体只提供**代码**而无名称：直接使用代码进行后续查询
- 如果代码和名称都齐全：直接进入 4.1.2

#### 4.1.2 数据获取
按以下顺序调用 MCP 工具（顺序执行，每只股票独立）：

1. **`stock_info`** — 基本信息
   - 调用：`mcp__aktools__stock_info(market=sh/sz/hk, symbol=代码)`
   - 返回字段：最新价、总市值、流通市值、行业、上市时间

2. **`stock_indicators_a`** — 财务指标（仅A股可用，港股跳过）
   - 调用：`mcp__aktools__stock_indicators_a(symbol=代码)`
   - 返回字段：净利润、净利润同比、毛利率、ROE、资产负债率、流动比率、存货周转天数

3. **`stock_prices`** — 技术指标（可选，增强风险提示）
   - 调用：`mcp__aktools__stock_prices(market=sh/sz/hk, symbol=代码, limit=5)`
   - 返回字段：MACD、KDJ、RSI、BOLL（取最新一日数据）
   - 注：若 `stock_info` 返回失败，此步骤跳过

#### 4.1.3 降级与重试
| 错误类型 | 处理方式 |
|----------|----------|
| 限流（429） | 等待5s，重试3次，间隔翻倍（5→10→20s） |
| 查不到（404/Not Found） | 页内标记 `⚠️ 数据不可用，待手动处理`，继续下一只 |
| 其他错误 | 同上 |

#### 4.1.4 写入实体页
获取市场数据后，直接将数据写入实体页的 `## 市场数据` 区块：
- 在步骤4创建实体页时，`## 市场数据` 区块已预置模板骨架
- 富化时填充实际数据（最新价、总市值、财务指标等）
- 若实体页已有数据，则更新（而非重复追加）
- 数据来源标注：`aktools | 更新时间：YYYY-MM-DD HH:mm`

#### 4.1.5 队列处理
- 多只股票时，顺序执行，不并发
- 每只股票独立重试（最多3次），失败则标记后跳过
- 全部完成后，输出汇总：`富化成功 X 只，失败 Y 只，待手动 Z 只`

- 步骤4完成后应有 `步骤4已完成=是`
- 步骤5完成后应有 `步骤5已完成=是`
若发现步骤进度记录与实际执行情况不符（如记录了步骤5完成但步骤4尚未执行），视为严重错误，**立即终止并回滚**。

在确认以下全部完成之前，本次 ingest 视为未完成。**必须通过工具调用验证，不能仅凭记忆或代码检查。**

1. **Sources 验证**：使用 `Glob` 确认 `wiki/sources/摘要-{slug}.md` 文件存在。
2. **Index 验证**：使用 `Read` 确认 `wiki/index.md` 已包含新增条目的文本。
3. **Log 验证**：使用 `Read` 确认 `wiki/log.md` 的最后一行是本次操作的记录，且时间正确。
4. **实体/概念验证**（如适用）：使用 `Glob` 确认相关页面已创建或更新。
5. **提取完整性**：对比步骤2记录的"应提取实体/概念清单"与步骤4实际创建的页面数量 
   - 如果清单有9项但只创建了0个实体/概念 → 强制失败，要求补充

#### 验证结果模板

只有以下清单全部打勾，才能输出最终结果：

```markdown
## ✅ Ingest 验证报告
- [ ] sources 页面已创建 (wiki/sources/摘要-xxx.md)
- [ ] 实体/概念页面已处理 (新增 X 个, 更新 Y 个)
- [ ] wiki/index.md 已追加新条目
- [ ] wiki/log.md 已追加记录 (时间: YYYY-MM-DD HH:mm)
- [ ] 原始文件未修改
```

**验证失败处理：**
如果上述任一检查失败，**必须回滚所有已写入的文件**（删除新增页面，撤销 index/log 的修改），并向用户报告失败原因。

验证完整后，询问用户是否继续处理下一个待处理的源文件。

## 冲突处理流程

当发现新旧知识冲突时（如定义矛盾）：

1. **暂停**：停止当前 ingest 流程。
2. **报告**：向用户说明冲突内容（哪个页面、冲突点是什么、新旧版本分别是什么）。
3. **⚠️ 确认检查点**：请用户选择处理方式：
   - A) 保留新旧两者，标注为"知识冲突"
   - B) 用新知识覆盖旧知识
   - C) 放弃本次 ingest
4. **继续**：根据用户选择继续或终止。

## 注意事项

- 所有 wiki 页面必须包含 `## 关联连接` 区域，不能产生孤岛页面。
- 使用简体中文编写所有内容。
- 实体命名使用 TitleCase（`Apple Inc.`），概念和来源使用 kebab-case（`growth-mindset`）。
- 绝对不修改 `raw/` 下的任何文件。
- 每次仅允许处理一篇 Markdown 文章，严禁批处理。
